# pad-lab

A small hands-on lab that mirrors the shape of [Community Tech Alliance's PAD stack](https://communitytechalliance.org/) — synthetic ActBlue donations flowing through GCS → BigQuery raw → dbt staging/mart.

**What this is:** A learning project I built to understand how PAD/PADdle works. Not production code, not affiliated with CTA.

**What this is not:** A claim of production data-engineering experience. I haven't operated BigQuery/Airbyte/dbt pipelines at PAD scale — this lab let me walk the layers and talk about them intelligently.

## Architecture

```
ActBlue CSV (simulated)
  → GCS landing bucket          # Airbyte would land vendor files here
  → pad_lab_raw                 # append-only, partitioned on created_at
  → pad_lab_staging (dbt view)  # clean, dedupe, type coercion
  → pad_lab_mart (dbt table)    # incremental daily aggregates
  → dashboards (SketchPAD/Looker in prod)
```

| Layer   | Resource                             | Production equivalent |
| ------- | ------------------------------------ | --------------------- |
| Landing | `gs://pad-lab-{project}/landing/`    | Airbyte → GCS         |
| Raw     | `pad_lab_raw.actblue_donations`      | PAD raw tables        |
| Staging | `pad_lab_staging.stg_donations`      | dbt staging models    |
| Mart    | `pad_lab_mart.daily_donation_totals` | dbt marts → SketchPAD |

## Prerequisites

- [gcloud CLI](https://cloud.google.com/sdk/docs/install) authenticated (`gcloud auth login`)
- Active GCP project with billing enabled
- Python 3.11+ (setup creates a local `.venv` with dbt-bigquery)

## Quick start

```bash
git clone <your-repo-url>   # or cd into this directory
cd pad-lab

# Bootstrap GCP resources + run dbt
./setup.sh

# Optional: override project
GCP_PROJECT=your-project-id ./setup.sh
```

## Key design choices

- **Partitioned raw table** on `created_at` — cost control exercise
- **Incremental mart** with merge on `(donation_date, campaign_id)` — simulates nightly dbt runs without full refresh
- **Batch 2 omits `gotv_march`** — intentional silent failure for data-quality debugging practice
- **dbt tests** on staging (unique `donation_id`) and mart (not_null on aggregates)
- **Dataset-scoped IAM** on a dedicated pipeline service account
